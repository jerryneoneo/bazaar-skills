# Autonomy & approval config — how much the agent does without asking

This is the **autonomy** setup: it lets the user run the agent hands-free so it can **list, search
(scan/cross-list), check chats** on the sell side and **search, offer, and close deals** on the buy
side without stopping to approve every step — while still guarding the few decisions that move real
money (an above-list bid on the sell side, an above-budget buy on the buy side). It has **two layers,
configured together** (see "Two layers of autonomy" at the end):

1. **Business approvals** — `data/config.json → approvals` (this file): per-decision gates.
2. **Harness permissions** — the Claude Code / Codex permission layer (`settings.local.json` allow
   list + permission mode), generated by `bin/install.py gen-settings --autonomy <level>`, so the
   agent isn't prompted for every tool call when running unattended.

Both are surfaced/edited from the `/bazaar` menu (`APPROVALS` anchor in
`skills/channel/onboarding.md`). The business-approvals layer replaces the old coarse
`autonomy_mode` (assist|auto) with a per-decision map.

## Schema (`data/config.json → approvals`)

```json
"approvals": {
  "preset": "balanced",
  "steps": {
    "listing_description": "auto",
    "listing_platforms":   "auto",
    "price_floor":         "confirm",
    "publish":             "auto",
    "buyer_replies":       "auto",
    "offers":              "confirm",
    "above_list_bids":     "escalate",
    "mark_sold":           "confirm",
    "distribution":        "confirm",
    "buy_search":          "confirm",
    "buy_offer":           "confirm",
    "buy_accept":          "confirm",
    "above_budget":        "escalate",
    "takeover":            "confirm"
  }
}
```

> The last four buyer-side keys (`buy_search`/`buy_offer`/`buy_accept`/`above_budget`) gate `/buy`,
> `skills/buying/search.md`, and `skills/buying/liaison-pipeline.md`. The example shows their
> `balanced` values; the user's chosen buy autonomy (e.g. "aggressive hands-free" →
> `buy_offer`/`buy_accept` = `auto`) is a normal per-step toggle and the live `config.json` reflects it.
> `takeover` is the **both-sides** gate for the inbox sweep (`skills/inbox-detect.md`): it asks before
> Bazaar steps into a conversation the user started on their own.

Each step value is one of:

| Value | Behavior |
|---|---|
| `auto` | Act without involving the seller. |
| `confirm` | Gate on `confirm()` (or a `notify(...)` with accept/decline); proceed only on yes. |
| `escalate` | Never auto. Always `notify()` the seller and park until they choose. |

### The fourteen steps

Sell side (9):

| Key | Decision point | Where it's read |
|---|---|---|
| `listing_description` | Use the drafted description as-is vs show it for edit | `skills/channel/listing.md` (draft step) |
| `listing_platforms` | Publish to the computed eligible platform set vs confirm the list | `skills/channel/listing.md` (PUBLISH) |
| `price_floor` | Accept suggested list price + floor vs confirm them | `skills/channel/listing.md` (awaiting_listing_inputs) |
| `publish` | Click publish vs confirm a preview first | `skills/channel/listing.md` + `skills/listing-flows/*.md` |
| `buyer_replies` | Send the composed answer to a question/shipping/availability | `skills/reply-pipeline.md` |
| `offers` | Negotiate a below/at-list offer vs surface it | `skills/reply-pipeline.md` (price_offer branch) |
| `above_list_bids` | An above-list / competitive-bid close | `skills/reply-pipeline.md` + `skills/channel/notifications.md` |
| `mark_sold` | Run the take-down recipe after a deal | `skills/channel/notifications.md` (sale → done) |
| `distribution` | Import an unmanaged listing / cross-list / set up a recommended platform | `skills/channel/distribution.md` (IMPORT, DISTRIBUTE) |

Buy side (4):

| Key | Decision point | Where it's read |
|---|---|---|
| `buy_search` | Run / refine a marketplace search for a stated want | `skills/buying/search.md` (searching step) |
| `buy_offer` | Send an opening / counter offer to a seller vs surface it | `skills/buying/liaison-pipeline.md` (counter_offer branch) |
| `buy_accept` | Commit a struck deal (within budget) + send the hand-off summary | `skills/buying/liaison-pipeline.md` (accepted branch) + `skills/buying/handover.md` |
| `above_budget` | Pursue / commit a listing priced above the secret max | `skills/buying/search.md` (confirm step) + `skills/buying/liaison-pipeline.md` |

Both sides (1):

| Key | Decision point | Where it's read |
|---|---|---|
| `takeover` | Step into an UNTRACKED inbox thread the user started solo — adopt the chat (set budget + liaise on the buy side, import the listing on the sell side) | `skills/inbox-detect.md` (TAKEOVER) |

**Hard floors (the "Harry guards"):** `above_list_bids` (sell), `above_budget` (buy), and `takeover`
(both) may never resolve to `auto`. Presets and the menu clamp them to `escalate` (or at most
`confirm`). `takeover` is hard-floored because stepping into a conversation the user is *personally*
in (the other party knows them as a human) is materially more sensitive than handling a fresh enquiry;
Bazaar always asks before adopting it. Once the user accepts, the *subsequent* per-message handling
reverts to the normal side gates (`buy_offer`/`buy_accept`, or `offers`/`buyer_replies`).
`bin/negotiate.py` returns `needs_seller_confirm: true` for any above-list offer, and the buyer engine
(`bin/buyer_negotiate.py` / `bin/budget_gate.py`) never emits an offer above the secret `max_budget`
(it climbs to `walk_away` instead) — so both gates are enforced in code regardless of config; the
config only controls *how* the edge case is surfaced.

## Presets

The menu resolves a preset to a full `steps` map:

| Step | hands-free | balanced (default) | all-steps |
|---|---|---|---|
| listing_description | auto | auto | confirm |
| listing_platforms | auto | auto | confirm |
| price_floor | auto | confirm | confirm |
| publish | auto | auto | confirm |
| buyer_replies | auto | auto | confirm |
| offers | auto | confirm | confirm |
| above_list_bids | escalate | escalate | escalate |
| mark_sold | confirm | confirm | confirm |
| distribution | auto | confirm | confirm |
| buy_search | auto | confirm | confirm |
| buy_offer | auto | confirm | confirm |
| buy_accept | auto | confirm | confirm |
| above_budget | escalate | escalate | escalate |
| takeover | confirm | confirm | confirm |

`balanced` reproduces today's behavior. The buyer who asked for **aggressive hands-free** buying runs
the buy rows at their `hands-free` values (`buy_offer`/`buy_accept` = `auto`) — the human still
executes the actual payment, which is the real money gate (this build coordinates payment, it does not
move money). Choosing a preset overwrites `steps`; the seller can then
toggle individual steps (which leaves `preset` as a label but the `steps` map is authoritative —
reads always use `steps`, never `preset`).

## The one read rule (used everywhere)

```
mode = config.approvals.steps[<key>]      # via the migration shim below if absent
if   mode == "auto":     act
elif mode == "confirm":  if confirm(<preview>): act   else: hold/skip
elif mode == "escalate": notify(<summary>, actions=[...]) ; park (resolve later)
```

## Close method (`seller_config.close.default_method`)

Separate from the approval gates above (which decide *whether* the agent acts), this decides *how* a
deal closes once the price is finalised. Read in `skills/reply-pipeline.md §3b` and
`skills/channel/notifications.md`:

| Value | Behavior |
|---|---|
| `"checkout"` | Auto-issue the carousell.ai **checkout link** (`bin/checkout.py`) and post it to the buyer. Payment + delivery are handled end to end — escrow, buyer protection, tracked shipping, zero seller fees. The hands-off, recommended path. |
| `"manual"` | **Full handover** — the agent posts a hand-off line, stops auto-replying (thread → terminal `handover`), and the seller arranges payment + delivery directly. |
| `null` / absent | **Ask the seller** at each close (a `close` escalation carrying `[checkout / manual]` actions); persist their pick here if they say "always". |

Checkout is the default we nudge toward because it is a **superior experience**, never a mandate —
`"manual"` is always available. The link is minted only at/above the hidden floor (`checkout.py`
re-validates via `floor_gate`; the floor value never appears). The rail itself (real escrow + label)
is a separate workstream — see the RAIL SEAM in `bin/checkout.py`.

## Migration shim (apply when `config.approvals` is absent)

Derive `steps` from the legacy keys so existing `config.json` keeps working until rewritten:

```
offers              = "auto"     if autonomy_mode == "auto"            else "confirm"
listing_description = "auto"     if listing_autonomy starts "auto"     else "confirm"
listing_platforms   = "auto"     if listing_autonomy starts "auto"     else "confirm"
publish             = "auto"     if listing_autonomy starts "auto"     else "confirm"
price_floor         = "confirm"
buyer_replies       = "auto"
above_list_bids     = "escalate"                          # always (close_gate bidding clause)
mark_sold           = "confirm"
distribution        = "confirm"                           # ask before importing / cross-listing
buy_search          = "confirm"                           # buyer-side gates: conservative if absent
buy_offer           = "confirm"
buy_accept          = "confirm"
above_budget        = "escalate"                          # always (mirror of above_list_bids)
takeover            = "confirm"                           # always (hard floor; never auto)
```

> **Absent newer keys (`distribution`, `buy_search`, `buy_offer`, `buy_accept`, `above_budget`,
> `takeover`):** if `approvals.steps` exists but has no entry for one of these, treat it as `"confirm"`
> (and `above_budget` as `"escalate"`). The distribution, buyer, and inbox-sweep flows never auto-act
> on a missing key — and `takeover` is never read as `auto` even if a stale config says so.

Legacy keys (`autonomy_mode`, `listing_autonomy`, `close_gate`) stay in `config.json` one release
for back-compat, then can be dropped. The numeric negotiation knobs are **unrelated** to approvals and
are never touched here. Of them, `max_counters` / `min_offer_ratio` / `lowball_cap` are now derived from
`data/style.json` `negotiation.sell_firmness` (see "Style / persona" below) and were removed from the
shipped default `config.json`; a value pinned in `config.json` still overrides the firmness preset.
`auto_counter_rounds` / `auto_counter_step` (per-item, via the floor record) and `price_anomaly_ratio`
(the listing anomaly check) are still read directly by `bin/negotiate.py` / the anomaly check.

## Style / persona (`data/style.json`)

Separate from *whether* the agent acts (approvals) and *how* a deal closes (close method): this is *how
it deals* — its voice and how firm it holds the line. Single source of truth, validated by
`bin/style.py`, rulebook in `skills/style.md` (folded into the cached prefix for the compose modes).

```json
{
  "voice": { "persona": "", "tone": "friendly", "humor": "light", "lowball_response": "polite" },
  "negotiation": { "sell_firmness": "balanced" },
  "learning": "suggest"
}
```

- **voice** shapes message *wording* at compose time (`skills/reply-pipeline.md` §4, buy-side
  `liaison-pipeline.md` §5, channel `say`/`ask`). `tone` ∈ friendly|warm|neutral|terse, `humor` ∈
  none|light|playful, `lowball_response` ∈ polite|firm|cheeky (how a `deflect_lowball` is worded, never
  a number), `persona` = free text. It never changes a decision, a number, or a `voice.md` invariant.
- **negotiation.sell_firmness** ∈ soft|balanced|firm|hardline derives the three sell knobs above
  (`bin/style.py FIRMNESS_PRESETS`; `balanced` == the old defaults, so the default profile is a no-op).
  `bin/negotiate.py _cfg()` resolves them as **explicit config knob > firmness preset > hard default**.
- **learning** ∈ off|suggest|auto gates the learning loop: `/pause` style steering
  (`skills/channel/corrections.md` §2e) and `/bazaar-eval` tone findings append proposals to
  `data/style_proposals.jsonl`; the user reviews/applies them under `/bazaar -> style`. Nothing rewrites
  the profile silently. Edited via `/bazaar -> style` (the `STYLE` onboarding anchor) or directly.

## Wake speed (how fast Bazaar notices a new message)

Separate from *whether* / *how* the agent acts: this is how fast it is **woken** by a new buyer
message. Each marketplace independently uses the fastest wake path actually available on this machine,
decided empirically by `bin/trigger_resolver.py` (`resolve(market) → "notification" | "poll"`), with
polling as the safe default. The user never sets a per-market flag; they choose one thing — whether to
grant Full Disk Access — and the resolver does the rest.

**Two modes, what to tell the user (pros only — see the no-cons rule):**

- **⚡ Instant** — turn on Full Disk Access. Bazaar replies the moment a buyer messages on a
  push-capable marketplace (Facebook / Instagram, Meta web push), often reading the message straight
  from the notification so it can answer without even opening the page. Fastest possible response.
- **🛡️ Standard** — no extra permissions, works the instant you install. Bazaar checks your inboxes on
  a quick cycle, fully hands-off, and never touches system notifications.

**How it actually works (operator detail):**

- `bin/notify_db.py` reads the macOS Notification Center DB (Chrome web-push notifications carry the
  source domain in the subtitle). It needs **Full Disk Access** on the daemon's process; without it,
  it fails open to empty and every market stays on **poll** (Standard).
- `bin/notify_watch.py` turns a new notification into a per-market trigger + content hint (idempotent
  via a rec_id cursor); wired into `agent_daemon.py` / `supervisor.py` through the `notify_trigger`
  seam, checked every loop iteration.
- A market is put on the **notification** path only when a readable notification from its origin has
  actually arrived (empirical, not a permission guess). So **Facebook/Instagram → Instant** once FDA
  is on; **Carousell → Standard/poll always** (no web-push subscription; browser-only, no backend).
- **Meta tabs only push while HIDDEN.** `bin/tab_park.py` keeps the marketplace tabs backgrounded in
  the dedicated warm Chrome between passes so their push keeps firing (a focused tab delivers in-app).
  This is automatic; poll-path markets (Carousell) read fine over CDP regardless of visibility.
- **Safety:** the notification path is fail-open everywhere — no FDA, DB drift, or a missing
  notification all degrade to polling. It never sends or mutates; it only decides *when to wake*.

To enable Instant: grant Full Disk Access to the daemon's Python (run `python3 bin/notify_setup.py
status` and grant FDA to `.python`). To go back to Standard: revoke it. No config write is required
either way — the resolver auto-detects. (An optional hard kill-switch knob can force poll even with
FDA on; not needed by default.) The daemon logs `⚡ wake mode: INSTANT` / `🛡️ STANDARD` at startup so
a restart confirms whether the grant took.

> **macOS gotcha (important):** FDA must land on a **Homebrew** Python, not `/usr/bin/python3`. The
> latter is the CommandLineTools shim: under launchd it re-execs a versioned framework binary that
> TCC won't attribute FDA to, so the grant never sticks and Instant silently stays off. `launchd/
> install_daemon.sh` therefore points the daemon at a Homebrew python (`brew install python` if
> absent); grant FDA to that interpreter.

## Two layers of autonomy (configured together)

Running hands-free needs both layers aligned — a `hands-free` business preset still stalls if the
harness prompts on every tool call, and a wide-open harness still asks the seller if the business
gates say `confirm`. The onboarding `APPROVALS` step (and `/bazaar → approvals`) sets both from one
choice:

| Autonomy level | Layer 1 — business approvals (`config.approvals.preset`) | Layer 2 — harness permissions (`install.py gen-settings --autonomy`) |
|---|---|---|
| **hands-free** | `hands-free` preset (auto list/search/replies; only `above_list_bids` escalates) | broad allow-list: Playwright + `Bash(python3 bin/*.py)` + `Bash(bin/*.sh)` + `Read/Write/Edit(data/**)` so unattended passes never prompt |
| **balanced** | `balanced` preset | same loop allow-list as hands-free (the gating that protects the seller is the business layer) |
| **all-steps** | `all-steps` preset (every step confirms) | narrower allow-list (no blanket `data/**` write) — the seller is present to approve |

- **Why two layers:** Layer 1 decides *whether Bazaar should act on its own*; Layer 2 decides
  *whether the harness lets the agent run a tool without a prompt*. The always-on daemon already
  passes Layer 2 via `run_pass.sh --allowedTools … --permission-mode acceptEdits`; for an
  interactive always-on loop (`/loop /sell-run`), Layer 2 comes from `settings.local.json`.
- **What hands-free unlocks (the user's goal):** on the sell side **list** (`publish`/`listing_*`),
  **search** (`distribution` scan + cross-list, plus comp research), and **check chats**
  (`buyer_replies`, below/at-list `offers`); on the buy side **search** (`buy_search`), **negotiate**
  (`buy_offer`), and **close within budget** (`buy_accept`) — all run without a prompt. Only an
  above-list bid (sell) or an above-budget buy (buy) ever stops to ask. This is what lets Bazaar run
  truly unattended.
- The `/bazaar` menu can still fine-tune individual business steps after picking a level; the
  harness layer only needs regenerating if the level changes.

## Follow-ups & listing health (proactive cadences)

Two background cadences that act on quiet threads and quiet listings. Both are deterministic at the
core (no LLM to *detect*; an LLM pass only *composes/sends*), default ON, and respect pacing caps,
quiet hours, and `/pause`. They surface in `/status` and `/bazaar-catchup` as **Follow-ups due** and
**Stale listings**.

**Stale-chat follow-ups** (`bin/followup_state.py`, ledger `data/followup_state.json`). When OUR last
message in a thread goes unanswered, nudge up to `followup_max_nudges` times on a gentle-escalation
cadence, then mark the counterpart "not interested". Applies to both buyers (sell) and sellers (buy).

| key | default | meaning |
|---|---|---|
| `followup_enabled` | `true` | master switch; `false` disables both nudges and drops |
| `followup_nudge_intervals_days` | `[1, 3]` | days of silence before nudge #1, then before nudge #2 |
| `followup_drop_after_days` | `3` | days after the last nudge before marking not interested |
| `followup_max_nudges` | `2` | how many nudges before giving up |
| `followup_poll_sec` | `3600` | how often the daemon CHECKS for due nudges/drops (cheap, file-only) |

The not-interested mark lives only in the follow-up ledger, never in the thread `status`, so a
counterpart who later messages back is still answered normally. Terminal threads (`lost`, `handover`,
`closed`, `agreed`, `held`) and threads with an open escalation are never nudged.

**Listing health** (`bin/listing_health.py`, ledger `data/listing_health_state.json`). A LIVE listing
with no buyer interest for `stale_days`+ gets concrete improvement suggestions on your channel (one
item per pass, deduped so you are not nagged).

| key | default | meaning |
|---|---|---|
| `listing_health_enabled` | `true` | master switch (suggestions go only to you, no outbound messages) |
| `stale_days` | `7` | days of no buyer inbound before a listing is "stale" |
| `listing_health_interval_hours` | `24` | min gap before starting a new stale-listing episode (rate-limit) |
| `rewarn_days` | `14` | min gap before re-warning the same listing |

All keys are read with these defaults, so an install with no entries behaves as above; set a key to
`0` (or `followup_enabled`/`listing_health_enabled` to `false`) to disable that behavior.
